# v0.2.0

Release date: 2026-02-27

## Highlights

- Major security hardening across the entire authentication and infrastructure stack.
- Smarter access request flow — the dashboard now guides users through requesting access automatically instead of showing generic errors.
- AWS region list is now complete (30 regions) and shows human-readable names in the selector.
- CloudWatch metrics now work correctly for all service panels under scoped sessions.

---

## Security fixes

### Critical

- **OTP codes are now cryptographically secure.** Previously generated using Python's standard `random` module (predictable under timing analysis). Now uses a CSPRNG (`secrets` module).
- **Legacy authentication bypass removed.** A secondary endpoint accepted raw AWS access keys directly, bypassing OTP verification and manager approval entirely. It has been removed.
- **Approval action hardened.** The approve/deny endpoint previously treated any value that wasn't exactly `"deny"` as an approval — including empty strings or typos. It now rejects anything other than `"approve"` or `"deny"` explicitly.
- **OTP endpoints are now rate limited.** Both the OTP request and verification endpoints were previously unlimited, enabling email flooding and brute-force attacks (6-digit = 1,000,000 combinations). Per-IP rate limiting is now enforced.

### High

- **Session cookies are now production-ready.** The `Secure` flag on session cookies was hardcoded to `false`. It is now configurable via `COOKIE_SECURE=true` for HTTPS deployments.
- **Backend API is no longer directly reachable from the host.** The API port was publicly exposed on the host, bypassing the nginx reverse proxy entirely. It is now only accessible within the internal Docker network.
- **HTML injection in notification emails fixed.** User-controlled values (names, denial reasons) were interpolated raw into email HTML. All email templates now properly escape user input.
- **OTP brute-force lockout added.** After 5 incorrect OTP attempts the code is invalidated and a new one must be requested, preventing exhaustive guessing within the 10-minute window.

## UX improvements

### Login flow
- Users with no active approval are automatically redirected to the Request Access form with their email already filled in — no more generic "Request failed" error.
- The standalone "Request Access" button has been removed. Entering your email on the login page handles both cases (login and new request) in one step.

### Request Access form
- The service list and duration options are now personalised. When you enter your email and move to the next field, the form immediately filters to only the services your admin has approved for you and the maximum duration you are allowed to request. You can no longer accidentally submit a request for services outside your allowlist.
- For brand-new users registering via the domain whitelist, all services and the full 12-hour duration are shown by default (admin can restrict after onboarding).

### Region selector
- The region dropdown now shows full human-readable names ("US East (N. Virginia)") instead of raw region IDs ("us-east-1").
- Region coverage expanded from 18 to 30 regions, including newer regions: Canada West, EU Zurich, EU Milan, EU Spain, AP Hong Kong, AP Hyderabad, AP Jakarta, AP Melbourne, AP Osaka, ME UAE, and Israel (Tel Aviv).

### Metrics
- CloudWatch metrics (CPU, network, disk, connections, etc.) now work correctly for all service panels (EC2, RDS, ElastiCache, MQ, OpenSearch, Load Balancers) under scoped STS sessions. Previously, metric charts were silently empty because the session policy was missing CloudWatch permissions.

## Configuration changes

The following variables from v0.1.0 are **removed** — remove them from your `.env` if present:

- `SECRET_KEY` — was defined but never used.

The following variables are **new** in v0.2.0:

- `COOKIE_SECURE` — set to `true` in production (HTTPS). Defaults to `false`.
- `CORS_ORIGINS` — comma-separated list of allowed origins. Defaults to `http://localhost:3000`.
- `VALKEY_PASSWORD` — password for the Valkey cache. Now required; Valkey runs unauthenticated without it.
- `OTP_EXPIRY_MINUTES` — how long an OTP code is valid. Defaults to `10`.
- `OTP_MAX_ATTEMPTS` — failed attempts before an OTP is invalidated. Defaults to `5`.

Updated minimal `.env` example:

```env
POWER_AWS_ACCESS_KEY_ID=...
POWER_AWS_SECRET_ACCESS_KEY=...
POWER_AWS_REGION=us-east-1
BASE_ROLE_ARN=arn:aws:iam::123456789012:role/PowerUserRole

ADMIN_EMAIL=admin@company.com

SMTP_HOST=smtp.company.com
SMTP_PORT=587
SMTP_USER=...
SMTP_PASSWORD=...
SMTP_FROM=noreply@company.com

POSTGRES_HOST=db
POSTGRES_PORT=5432
POSTGRES_DB=awsdashboard
POSTGRES_USER=awsdashboard
POSTGRES_PASSWORD=<strong-password>

VALKEY_PASSWORD=<strong-password>
VALKEY_URL=redis://:${VALKEY_PASSWORD}@valkey:6379

COOKIE_SECURE=false
CORS_ORIGINS=http://localhost:3000
OTP_EXPIRY_MINUTES=10
OTP_MAX_ATTEMPTS=5
```
